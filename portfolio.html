Overview

何を、なぜ作ったか

従来のファン情報は各所に分散し、「知識の羅列」になりがち。それを、曲を中心に据えた一つの構造として組み直すのが目的だった。

課題

情報の分散

楽曲・原作の文学・聖地・考察が、別々のサイトにばらばらに存在する。探すのに知識と手間が要る。

方針

量ではなく構造で

網羅の競争をやめ、曲を「感情×時間」の空間に置く。まず少数を丁寧に設計してデータ構造を固め、全曲へ拡張した。

到達点

育つデータベース

約110曲を網羅。世界観マップ・楽曲・聖地アトラス・年表の4つの見方で、同じデータを多面的に読める。

How it works

曲を、空間に置く

曲データ → 座標 → 描画。表示の難所は「多すぎる星」と「地図のAPIキー」。それぞれに一つずつ設計判断がある。

01 — 配置

感情 × 時間の2軸へ

各曲に、時間 t（朝→夜）と感情 e（静寂→高揚）の座標を与える。星の位置は割合で置くだけ。色は季節。

t = 0..1 // 朝 → 夜 e = 0..1 // 静寂 → 高揚 left = 5 + t·90 % top = 7 + (1−e)·84 %

02 — ズーム

110の星が、重ならない

全体を拡大すると星も曲名も一緒に巨大化して潰れる。そこで星の中身だけを逆スケールし、画面上のサイズを一定に保つ。拡大＝星が離れるだけで、文字は重ならない。

inner.transform = scale(cz) // 全体を拡大 star.transform = scale(1/cz) // 中身は逆スケール

03 — 地図

キー不要の聖地マップ

聖地（MVのロケ地・原作の舞台・歌詞の実在地名）は地図に置く。Google Maps はキーと課金設定が要り、単一HTMLだとキーが露出する。Leaflet + OpenStreetMap を選び、キーなし・無料で地図を成立させた。日本各地に加え、二部作の舞台スウェーデンまで含む。

Data architecture

手作業の丁寧さと、自動化の広さ

Constraint as design

著作権を、設計に変える

歌詞は載せられない。この制約を回避ではなく、設計判断として扱った。

歌詞は、メタデータにする

歌詞本文はアプリに一切載せず、外部の公式・許諾サイトへリンク。歌詞は解析して概念タグ（単語）に変換する形でのみ活用し、本文はリポジトリにも残さない。原作の出典は作者・作品名という事実として示す。

信頼性を、混ぜない

考察の確度を3層に分離し、常に色で区別する。「本人が公言した事実」と「ファンの解釈」を同じ顔で並べない ── 情報の誠実さをUIの第一原則にした。

公式

本人の言及・原作の明確な一致・タイアップ

有力考察

広く共有され、根拠のある読み

ユーザー

個人的な解釈の余白

Design Principles

設計思想

情報を並べることより、テーマを「歩ける構造」に翻訳することを優先した。

01

構造で勝つ

量では既存サイトに勝てない。曲を空間に配置し、関係を見せることで「体験」の価値を作る。

02

継続的に育てられる形に

データ層と表示層を分離し、注入レイヤーに1行足すだけで全曲へ反映。生きたDBとして更新し続けられる。

03

制約を、設計に

著作権もAPIキーも、回避ではなく設計に変える。歌詞→概念タグ、地図→キー不要のLeaflet。

04

見える不満を、ロジックに翻訳する

「拡大で文字が重なる」不満を、星の逆スケールという式で根本解決。演出でごまかさない。

Learnings

得られた学び

  • 情報設計：信頼度のラベリングと3層分離で、誠実さをUIに落とす。
  • データパイプライン：核と自動生成を分け、後付けで育てられる構造を作る。
  • 制約の設計化：著作権・APIキーを、回避でなく判断に変える。
  • 一次情報の裏取り：公式サイト・本人発言まで遡り、考察と事実を切り分ける。
